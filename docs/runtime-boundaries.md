# 运行边界

## 项目定位

当前仓库提供的是一个轻量 WARP SOCKS5 出口，核心路径是：

1. 选择注册后端：`teams`、`wgcf-free`、`wgcf-plus`
2. 生成并复用 `./wireguard` 下的运行状态
3. 拉起 `wg0`
4. 通过 `microsocks` 暴露 SOCKS5

它不是以下方案的等价实现：

- Cloudflare 官方当前 `Local proxy mode`
- 官方 `warp-cli registration license` 容器化封装
- 带自动修复、主机网络治理或多协议代理的完整平台

## 已做与未做

当前仓库已经做的事情：

- 支持三种注册后端
- 支持持久化状态复用
- 支持端点临时覆盖
- 提供极简 `HEALTHCHECK`

当前仓库刻意不做的事情：

- 不引入 `warp-svc`、`warp-cli`、`dbus`
- 不提供 HTTP/UDP/ICMP 多协议代理
- 不自动修改宿主机 `nft` / `ip rule`
- 不做自动恢复或自愈逻辑

## 端口暴露风险

`microsocks` 是无认证 SOCKS5 代理。

默认配置把端口绑定到 `127.0.0.1`，这是推荐的安全边界。如果你把 `HOST_BIND_IP=0.0.0.0`，等价于把未鉴权代理对外暴露。只有在下面条件都满足时，才应考虑这样做：

- 你明确知道谁会访问这个端口
- 宿主机或上游网络已经有额外 ACL / 防火墙限制
- 你接受该端口被误用后带来的出口滥用风险

如果只是本机程序使用代理，不要改成 `0.0.0.0`。

## 健康检查边界

当前 `HEALTHCHECK` 只做一件事：通过容器内本地 SOCKS5 访问 `https://cloudflare.com/cdn-cgi/trace`，检查是否返回 `warp=on` 或 `warp=plus`。

这意味着它只回答“当前出口是否可用”，不回答以下问题：

- 为什么不可用
- 是否应该自动重试或切换端点
- 是否应该修改宿主机路由

如果后续需要自动恢复，应把它设计为独立、可选能力，而不是塞回启动主链路。
